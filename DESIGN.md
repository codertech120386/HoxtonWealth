# Design Document — Hoxton Wallet & Ledger Platform

This document explains the *why* behind the implementation. The README covers *how* to run it; this covers the engineering judgment behind the schema, the processor, the failure model, and the path to scale.

---

## Reviewer's guide — how the six evaluation criteria are addressed

The exercise spec lists six dimensions for evaluation. This section maps each one to a concrete answer plus the file or doc-section a reviewer should open to verify it.

### 1. Data modelling for financial systems

**How addressed.** Append-only `ledger_entries` table with NOT NULL parent FK to `transfers` — every ledger row has a parent operation, so reconciliation is unambiguous. Balances are computed from the ledger on every read (no balance column to drift). "Transfer" is the universal value-movement object — deposits are modelled as transfers from the system account, so a single state machine and idempotency mechanism cover both flows. UUIDv7 PKs for B-tree locality on an append-heavy workload. DB-layer integrity throughout: CHECK constraints on enums and amounts, `transfers.idempotency_key` UNIQUE, partial unique index `accounts_one_system` so the database itself rejects a second system account.

**Where to look.**
- `database/migrations/` — the four core tables.
- `app/Models/` — Eloquent casts that translate VARCHAR + CHECK columns into PHP backed enums.
- DESIGN.md **§2** (full rationale, including why ledger-based balances and why deposits-as-transfers).

### 2. Failure handling and system resilience

**How addressed.** Two-phase transfer processor: Phase 1 commits `PROCESSING` status + audit row in its own transaction so the lifecycle stays observable; Phase 2 is an atomic settlement (lock both accounts via `SELECT ... FOR UPDATE` ordered by UUIDv7 ASC for deadlock-freedom, recompute balance under the lock to close the TOCTOU window, post double-entry, mark COMPLETED — all-or-nothing). Business-rule failures (insufficient balance) → `markFailed()` + `$this->fail()` so Laravel does not retry. Transient failures → re-thrown so the framework retries with exponential backoff up to `tries=3`; final failure lands in `failed_jobs` (DLQ). Crash mid-Phase-2 leaves no partial ledger writes; a `transfers:retry-failed` artisan command replays from the DLQ. `ledger:reconcile` independently verifies double-entry + zero-sum + no-negative-user-balance invariants from outside the application.

**Where to look.**
- `app/Services/TransferProcessor.php`, `app/Jobs/ProcessTransferJob.php` — the two-phase implementation.
- `tests/Feature/Transfers/ProcessTransferJobTest.php` — `crash_mid_settlement`, `retry_after_crash`, `two_sequential_transfers_from_the_same_account_never_overdraw_it`, `insufficient_balance`.
- `app/Console/Commands/RetryFailedTransferCommand.php`, `app/Console/Commands/ReconcileLedgerCommand.php`.
- DESIGN.md **§3** (event flow), **§4** (failure handling matrix).

### 3. Event-driven architecture thinking

**How addressed.** A fixed six-event vocabulary (`AccountCreated`, `DepositMade`, `TransferRequested`, `TransferProcessing`, `TransferCompleted`, `TransferFailed`) modelled as a PHP backed enum and pinned in the DB by a CHECK constraint on `audit_logs.event_type`. `audit_logs` is the durable event log — every state-changing service writes its event row inside the same DB transaction as the state change, so the event log and the application state can never disagree. Events are the unit of correlation: `correlation_id` (UUIDv7) generated by middleware on the API side, propagated through the `ProcessTransferJob` constructor into the worker, pushed onto Log context inside `handle()`, written onto every audit row. Reconstructing a transfer's full lifecycle is `WHERE correlation_id = ?` — the same ID joins HTTP logs, worker logs, and the audit table.

**Where to look.**
- `app/Enums/AuditEventType.php` — the vocabulary.
- `app/Http/Middleware/CorrelationId.php`, `app/Jobs/ProcessTransferJob.php` — propagation.
- audit_logs writes inside `AccountService`, `TransferService`, and `TransferProcessor` (always inside the same `DB::transaction` as the state change).
- DESIGN.md **§3** (event flow lifecycle), **§6.8** (audit & non-repudiation).

### 4. API design and idempotency

**How addressed.** Every state-changing endpoint takes an `idempotency_key`, which becomes a UNIQUE column on `transfers`. Idempotent replay returns the original row — semantically distinguished from first-success at the HTTP layer (deposits: 201 first, 200 replay; transfers: 202 first, 200 replay). The pattern survives concurrent retries because the service catches `UniqueConstraintViolationException` and re-fetches the race winner's row. Beyond idempotency: cursor pagination (not offset) on ledger history for stable pages; clear HTTP semantics (401/404/422/429 each have a single meaning); `ForceJson` middleware so error responses are always JSON regardless of `Accept` header; correlation ID returned on every response (including 401s) for client-side debugging; full OpenAPI annotations on every endpoint, browsable at `/api/documentation`.

**Where to look.**
- `app/Http/Controllers/Api/V1/` — eight controller methods, each with `OA\*` attribute documenting all response codes.
- `app/Services/TransferService.php` — `deposit()` and `initiateTransfer()` show the idempotency pattern with race handler.
- `tests/Feature/Deposits/DepositTest.php`, `tests/Feature/Transfers/InitiateTransferTest.php` — verify idempotent replay produces no duplicate rows / jobs.
- Swagger UI at `http://localhost:8080/api/documentation`.
- DESIGN.md **§6.3** (replay & idempotency).

### 5. Security and auditability awareness

**How addressed.** Static API key with SHA-256 hash + `hash_equals` constant-time comparison; mass-assignment guard (`is_system` not in `$fillable`) so accounts can't be promoted via the API; system account invisible from API surface (404 on direct access); per-API-key rate limit on transfer initiation with separate buckets and `Retry-After` header; append-only audit trail with fixed event vocabulary and correlation IDs; structured JSON logs with the same correlation IDs flowing through HTTP and worker; defense in depth at the DB layer (CHECK constraints, FK integrity, partial unique on system account); `ForceJson` middleware to prevent HTML error pages leaking framework chrome. Section 6 of this document gives a full threat-model treatment with explicit lists of (a) what's defended against, (b) deliberate production-hardening trade-offs, (c) classes deliberately out of scope, (d) residual gaps an auditor would surface.

**Where to look.**
- DESIGN.md **§6** (the full 14-subsection security treatment).
- `app/Http/Middleware/ApiKeyAuth.php`, `app/Http/Middleware/CorrelationId.php`, `app/Http/Middleware/ForceJson.php`.
- `app/Providers/AppServiceProvider.php` — RateLimiter definition keyed on the hashed API key.
- `database/migrations/` — CHECK constraints, partial unique indexes.
- `tests/Feature/RateLimit/RateLimitTest.php`, `tests/Feature/Logging/LoggingTest.php`.

### 6. Engineering maturity and trade-offs

**How addressed.** The project was scoped to the spec's "do not over-engineer" instruction, with every choice point logged in PLAN.md and trade-offs surfaced in three places — README "Trade-offs", DESIGN §2 (schema), §6.12 (security trade-offs), and §7 (deliberate non-features). Real correctness was prioritised over surface-level completeness: the reconciliation command and the DLQ replay command exist because they catch problems we can't be sure don't exist; the two-phase processor was chosen over the simpler one-tx variant because debuggability of crashes matters more than minimising commits. Honest gaps are documented rather than papered over — the stuck-`PROCESSING` recovery sweeper, forking-process concurrency tests, and per-key signing are all explicitly called out as production-hardening work rather than silently omitted. Tests prove behaviour (89 tests, ~398 assertions) rather than padding coverage. AI usage is disclosed honestly with the actual prompts that shaped key decisions.

**Where to look.**
- README "Trade-offs" section.
- DESIGN.md **§2** (data-model trade-offs), **§5** (scaling), **§6.12** (security trade-offs), **§7** (deliberate non-features).
- AI_USAGE.md — disclosure including which prompts shifted the design.
- PLAN.md — full per-task plan kept in-repo for transparency about what got built when.

## 1. Architecture

### 1.1 Component map

```
┌──────────────────────────────────────────────────────────────────┐
│  HTTP API (api/v1/*)                                              │
│  ┌────────────────────────────────────────────────────────────┐  │
│  │ ForceJson  →  CorrelationId  →  ApiKeyAuth  →  Controller │  │
│  └────────────────────────────────────────────────────────────┘  │
│       │                                                           │
│       ├─ AccountController.store        ──┐                       │
│       ├─ AccountController.show           │                       │
│       ├─ LedgerController.index           │                       │
│       ├─ DepositController.store    ──────┼── Application services
│       ├─ TransferController.store         │   (AccountService,    │
│       │   (throttle:transfers)            │    LedgerService,     │
│       └─ TransferController.show       ───┘    TransferService)   │
│                                           │                       │
└───────────────────────────────────────────┼───────────────────────┘
                                            ▼
                                ┌───────────────────────┐
                                │  Postgres 16          │
                                │   accounts            │
                                │   transfers           │
                                │   ledger_entries      │
                                │   audit_logs          │
                                │   jobs / failed_jobs  │
                                └───────────────────────┘
                                            ▲
                                            │
                            ┌───────────────────────────────┐
                            │  worker (php artisan         │
                            │  queue:work, restart:        │
                            │  unless-stopped)             │
                            │                              │
                            │  ProcessTransferJob ─▶       │
                            │      TransferProcessor       │
                            │        (Phase 1 + Phase 2)   │
                            └───────────────────────────────┘
```

### 1.2 Process boundaries

Four containers in `docker-compose.yml`:

- **app** (php-fpm) — handles HTTP requests via the `web` service.
- **web** (nginx) — terminates HTTP on `:8080`, forwards PHP to `app:9000`.
- **worker** (php cli running `queue:work`) — picks up `ProcessTransferJob` from the `jobs` table.
- **db** (postgres) — single source of truth for accounts, ledger, audit, queue.

Splitting `app` and `worker` into separate containers (sharing the same image) lets us scale them independently and crash one without taking down the other.

### 1.3 Why these layers

- **Controllers stay thin.** They validate input via FormRequests, resolve account UUIDs, and delegate to a service. No business logic lives in controllers.
- **Services own the writes.** `TransferService` and `AccountService` are the only things that mutate state. Each method that writes wraps its work in a `DB::transaction`.
- **`TransferProcessor` is split out from the job.** The job is a thin queue-framework adapter (handles retry policy, correlation logging, fail-vs-retry decisions). The processor is pure domain logic. This split makes the processor unit-testable without queue infrastructure — every concurrency / crash test runs the processor directly.
- **`audit_logs` is the event log.** We don't ship a separate event store. Every state-changing service writes an audit row in the same DB transaction as the state change, so the audit trail can never disagree with the ledger.

## 2. Data Modelling

### 2.1 Schema (4 application tables)

```
accounts
  id          UUIDv7 PK
  name        VARCHAR
  is_system   BOOL NOT NULL DEFAULT false
  timestamps
  CREATE UNIQUE INDEX accounts_one_system ON accounts (is_system) WHERE is_system = true

transfers
  id              UUIDv7 PK
  type            VARCHAR  CHECK (type IN ('TRANSFER','DEPOSIT'))
  idempotency_key VARCHAR UNIQUE
  from_account_id UUID FK accounts(id)
  to_account_id   UUID FK accounts(id)
  amount          BIGINT  CHECK (amount > 0)
  status          VARCHAR CHECK (status IN ('PENDING','PROCESSING','COMPLETED','FAILED'))
  error_reason    TEXT NULL
  attempts        INT NOT NULL DEFAULT 0
  timestamps
  index (status), (from_account_id), (to_account_id), (type, created_at DESC)

ledger_entries
  id          BIGSERIAL PK
  account_id  UUID FK accounts(id)
  transfer_id UUID FK transfers(id)  NOT NULL
  direction   VARCHAR CHECK (direction IN ('DEBIT','CREDIT'))
  amount      BIGINT  CHECK (amount > 0)
  created_at
  index (account_id, created_at)

audit_logs
  id              BIGSERIAL PK
  event_type      VARCHAR CHECK (event_type IN (
                    'AccountCreated','DepositMade',
                    'TransferRequested','TransferProcessing',
                    'TransferCompleted','TransferFailed'))
  account_id      UUID FK accounts(id)  NULL
  transfer_id     UUID FK transfers(id) NULL
  correlation_id  UUID NOT NULL
  payload         JSONB
  created_at
  index (transfer_id, created_at), (account_id, created_at), (correlation_id)
```

### 2.2 Why ledger-based balances (not a balance column)

The exercise spec is explicit: "directly mutating balances without a ledger" is a forbidden practice. There's a deeper engineering reason though:

- **Auditability is structural, not bolted-on.** The ledger *is* the source of truth. Reconstructing what happened to an account is `SELECT * FROM ledger_entries WHERE account_id = ?` — no cross-table joins, no derived state to second-guess.
- **Concurrency story is simpler.** The only writer to `ledger_entries` is the inside of a `DB::transaction` that locked both involved accounts via `SELECT ... FOR UPDATE`. There's no "balance was updated by someone else" race because there's no balance column to race on.
- **Reconciliation becomes straightforward.** A separate process recomputes balance from the ledger and compares. We do this in the `ledger:reconcile` artisan command (§4.4).

The trade-off — O(N) read cost per balance lookup — is real but not interesting at this scope. In production we'd add a `account_balance_snapshots` table updated atomically with each ledger write; the live ledger remains the truth, the snapshot is a fast cache. We didn't build that here because it's a known pattern with no novel design questions, and the spec emphasizes correctness over performance.

### 2.3 Why deposits are modelled as a kind of transfer

Naively, deposits and transfers are different objects: a deposit moves money from "outside the system" to a user; a transfer moves money between two users. We could put them in two tables (`transfers` + `deposits`) with their own state machines.

We didn't, because every ledger row needs a parent operation it can FK to — otherwise `ledger_entries.transfer_id` has to be nullable, and "this row has no parent" is a smell that ripples through reconciliation, audit queries, and ledger history responses. The fix is to broaden the definition of "transfer" to mean *any movement of value between two accounts, one of which may be the system account*. Deposits become `type=DEPOSIT` rows whose `from_account_id` is the system account. Then:

- `ledger_entries.transfer_id` is **NOT NULL** — every entry has a parent.
- Idempotency, status, audit trail, ledger-history queries all work uniformly across the two paths.
- Adding withdrawals later needs only a third `type` value plus a CHECK constraint update; no new tables.

The cost is a small terminological stretch ("transfer" includes "deposit"), which is pure cosmetics.

### 2.4 Why deposits are synchronous while transfers are asynchronous

Structurally, a deposit *is* a transfer (system account → user account, same `LedgerService::postDoubleEntry` posting). So why does `POST /accounts/{id}/deposits` settle inline and return 201 with a balance, while `POST /transfers` returns 202 and hands work to a queue?

Four reasons, in decreasing order of weight:

- **Failure surface is trivial for deposits.** A user→user transfer can fail on a real business check (insufficient funds in the sender). That failure is worth tracking through `PENDING / PROCESSING / COMPLETED / FAILED`, retrying, and DLQ-replaying. A deposit cannot fail that way: the system account is the source of money and is treated as unbounded. The only failure modes are "account doesn't exist" (rejected at the controller before any state change) or DB outage. There's no state machine worth the latency.
- **Lock contention is bounded for deposits.** `TransferProcessor::process()` locks two arbitrary user accounts; under load, ordering pairs of locks is the classic deadlock setup, and the queue + UUIDv7-ordered locking gives us a controlled answer (§3.3). A deposit only ever touches one user account plus the singleton system account — contention is bounded by the schema itself, not by careful lock ordering.
- **Caller UX matters for deposits.** The deposit response returns the new balance (`DepositController::store`). Async would force the caller to poll `GET /transfers/{id}` for an op that essentially never fails. A transfer's `202 Accepted` + status polling is acceptable because both parties can wait; an interactive deposit shouldn't.
- **Real-world modelling.** A deposit represents money arriving from an external rail (card, bank) where authorization has already happened upstream — the API call is just recording it. An internal user→user transfer is the system itself authorizing, so that authorization work is what gets queued.

We could make deposits async with the same `Transfer` row plus a `PENDING → COMPLETED` step, and `GET /transfers/{id}` already supports both types. It just wouldn't earn its keep — none of what async buys (failure handling, deadlock avoidance, queue smoothing, retry) applies to a single-lock, no-balance-check operation.

### 2.5 Why the system account exists, and why `is_system`

Double-entry's invariant is `SUM(signed amount) == 0` across the entire database. So when a user deposits $100, *something* has to be debited the matching $100 — otherwise we've conjured money out of thin air and the global sum drifts. The conventional answer is to model "outside the platform" as an internal counterparty account. Banks call it variously: *world*, *external*, *issuance*, *cash-on-hand*. Ours is just `is_system = true`.

`is_system` is a boolean rather than a magic UUID in config because:

- The schema self-documents — anyone running `\d accounts` immediately sees that some rows are special.
- The partial unique index (`UNIQUE(is_system) WHERE is_system = true`) lets the database itself enforce "exactly one system account exists". A magic-UUID-in-config approach has no such guarantee.

The system account is bootstrapped by `database/seeders/SystemAccountSeeder.php` using `firstOrCreate(['is_system' => true], ['name' => 'system'])`, so running migrations + seed is idempotent: re-seeding never creates a second one, and the partial unique index would reject it anyway.

### 2.6 Why UUIDv7 for entities, BIGSERIAL for log tables

The schema uses two different ID strategies, picked deliberately:

- **`accounts.id`, `transfers.id`** — UUIDv7. These rows are addressable from outside the system: account UUIDs flow through API URLs, idempotency requires globally unique transfer IDs across retries that may originate on different machines, and the worker dispatches jobs by transfer ID before the row has propagated through any monotonic counter.
- **`ledger_entries.id`, `audit_logs.id`** — `BIGSERIAL`. These rows are append-only internal log entries. They're never API-addressable (you query them by `account_id` / `transfer_id` / `correlation_id`, never by their own ID), they're write-heavy and benefit from the cheapest possible insert, and a sequence gives us a free total ordering useful for cursor pagination on ledger history. UUIDs would buy nothing here.

The mixed strategy is the same reasoning the database itself uses: `pg_class` rows have an `oid`, but `pg_stat_activity` rows don't.

**Why UUIDv7 specifically (not UUIDv4).** UUIDv7's high 48 bits are a Unix-millisecond timestamp. So new rows append to the right edge of any B-tree index instead of fragmenting it (the way UUIDv4 does). For an append-heavy ledger workload that's a big difference: better page-cache locality, faster range scans on "recent transfers", less index bloat. The Postgres column type stays `uuid` — only the generator changes.

**Note on Postgres compatibility.** UUIDv7 is a generation algorithm (RFC 9562), not a database feature. Postgres has stored arbitrary 128-bit values in the `uuid` type since version 8.x and does not parse, validate, or care about the version digit. We generate the value in PHP via `Str::uuid7()` (in `App\Concerns\HasUuidv7s`) and pass the bytes to Postgres on INSERT — no extension, no minimum version. Postgres 18 (released Sep 2025) added a native `uuidv7()` SQL function for server-side generation; we don't use it because we generate IDs in the app layer, so Postgres 16 is sufficient.

Trade-off: UUIDv7 leaks creation time. Acceptable for our domain since timestamps are public anyway.

### 2.7 Why VARCHAR + CHECK rather than native Postgres ENUM

`ALTER TYPE … ADD VALUE` is one-way (no remove or rename without surgery), and Laravel migrations don't model it cleanly. A CHECK constraint is portable, evolvable in a single ALTER, and pairs well with PHP 8.1 backed enums in models for compile-time safety. The cost is a small column-storage difference, negligible at our scale.

### 2.8 What `audit_logs` carries (and what it does not)

`audit_logs` is **append-only by convention** — no setters on the model, no UPDATE / DELETE paths in service code. Its `event_type` vocabulary is fixed at six values via a CHECK constraint:

| `event_type` | Emitted when |
|---|---|
| `AccountCreated` | `POST /accounts` succeeds |
| `DepositMade` | `POST /accounts/{id}/deposits` succeeds |
| `TransferRequested` | `POST /transfers` accepts a new request |
| `TransferProcessing` | `ProcessTransferJob` enters Phase 1 |
| `TransferCompleted` | Phase 2 commits |
| `TransferFailed` | Job sets status=FAILED |

We deliberately picked **explicit nullable FKs** (`account_id`, `transfer_id`) rather than a polymorphic `aggregate_type` + `aggregate_id` pair, because:

- Real referential integrity (FK constraints catch typos).
- Easier indexing and queries (`WHERE transfer_id = ?` directly).
- We only ever have two aggregate types in this domain — the polymorphic flexibility wasn't earning its keep.

`correlation_id` carries through HTTP request → audit row → background job → audit rows written by the job, so reconstructing a single logical operation is a one-column WHERE clause.

## 3. Event Flow

### 3.1 The two-phase transfer processor

This is the core of the system, and the most subtle piece. `ProcessTransferJob::handle()` delegates to `TransferProcessor::process()`, which runs **two committed transactions**:

```
                ┌─────────────────────────────────────────────────────────┐
                │  Phase 1 (own committed tx)                              │
                │    UPDATE transfers SET status = PROCESSING              │
                │    INSERT INTO audit_logs (event_type=TransferProcessing)│
                │  COMMIT                                                  │
                └─────────────────────────────────────────────────────────┘
                                         │
                                         ▼
                ┌─────────────────────────────────────────────────────────┐
                │  Phase 2 (own committed tx)                              │
                │    SELECT * FROM accounts                                │
                │      WHERE id IN (from_id, to_id)                        │
                │      ORDER BY id ASC                                     │
                │      FOR UPDATE                       ← deadlock-free    │
                │                                         (UUIDv7 ordering)│
                │    sender_balance := SUM(ledger_entries) UNDER LOCK      │
                │                                                          │
                │    if sender_balance < amount:                           │
                │       throw InsufficientBalanceException  (rolls back)   │
                │                                                          │
                │    INSERT 2 ledger_entries (debit + credit, same amount) │
                │    UPDATE transfers SET status = COMPLETED               │
                │    INSERT INTO audit_logs (event_type=TransferCompleted) │
                │  COMMIT                                                  │
                └─────────────────────────────────────────────────────────┘
```

### 3.2 Why two phases (rather than one big transaction)

A single transaction is simpler — but a Phase-2 rollback in the single-tx design erases all evidence the worker ever started, so the transfer flips back to `PENDING` and is indistinguishable from "never picked up". When you're trying to debug why a transfer is stuck, that ambiguity is exactly what you don't want.

Splitting Phase 1 out as its own commit means:

- A `PROCESSING` transfer with no terminal event is unambiguously "we crashed mid-handle".
- The `TransferProcessing` audit row is preserved on every attempt, so the audit trail shows multiple attempts as multiple `TransferProcessing` events leading up to a final terminal event.
- Insufficient-balance failures still get a clean `[Processing, Failed]` audit chain because Phase 1 committed before Phase 2 rolled back.

The cost: a transfer can be left stuck in `PROCESSING` forever if it exhausts retries before completing. We address that operationally — see §4 — but a "sweep PROCESSING transfers older than N minutes" command is the obvious next addition.

### 3.3 Why we lock both accounts in UUIDv7-ascending order

Two concurrent workers settling transfers on overlapping account pairs can deadlock if they grab locks in different orders. Sorting by UUIDv7 is a classic deadlock-avoidance pattern: every worker locks accounts in the same global order, so circular wait is impossible.

UUIDv7's timestamp ordering is irrelevant here — any total order would work, but we use `id` because it's already the primary key.

### 3.4 Why we recompute balance under the lock

Reading sender balance *before* acquiring the lock would create a TOCTOU window:

```
read balance      → 1000 (looks fine for a 800 transfer)
                  ← someone else commits a 500 debit
acquire lock
write 800 debit   → balance is now -300
```

By recomputing after `lockForUpdate()` blocks until any prior writer commits, we guarantee the balance we check is the same balance we then debit. If two transfers race for the same account, the first wins, the second sees the new (post-debit) balance and is rejected with `InsufficientBalanceException`. The `ProcessTransferJobTest` explicitly verifies this with `two_sequential_transfers_from_the_same_account_never_overdraw_it`.

### 3.5 Lifecycle states (visible to API consumers)

```
                       ┌──────────┐
   POST /transfers ───▶│ PENDING  │
                       └────┬─────┘
                            │ worker picks up the job
                            ▼
                       ┌─────────────┐
                       │ PROCESSING  │ ← Phase 1 committed
                       └────┬────────┘
                  ┌─────────┴─────────┐
        Phase 2   │                   │   Insufficient balance
        commits   ▼                   ▼   (or retry exhausted)
                ┌────────────┐    ┌──────────┐
                │ COMPLETED  │    │  FAILED  │
                └────────────┘    └──────────┘
```

Deposits go straight from request to `COMPLETED` synchronously — they have no `PENDING` or `PROCESSING` phase. See §2.4 for why the two flows are intentionally asymmetric.

## 4. Failure Handling

### 4.1 Scenarios handled

| Scenario | Detection | Recovery |
|---|---|---|
| **Worker crash mid-Phase-2** | Postgres detects connection drop, rolls back the in-flight tx | Transfer stays at `PROCESSING` (Phase 1 committed). Job remains visible to the queue; on re-attempt, runs Phase 1 again (idempotent — UPDATE no-op + new audit row), then Phase 2 fresh. Tested by `crash_mid_settlement` and `retry_after_crash` |
| **Duplicate transfer request** | UNIQUE constraint on `transfers.idempotency_key` | First request commits the transfer; second request's INSERT loses the race, gets `UniqueConstraintViolationException`, returns the existing row with HTTP 200 instead of HTTP 202 |
| **Insufficient balance** | Recompute balance under `FOR UPDATE` lock; throw `InsufficientBalanceException` | `markFailed()` writes status=FAILED + `error_reason` + `TransferFailed` audit row, then `$this->fail()` so Laravel does NOT retry. Balances unchanged. Tested by `insufficient_balance` |
| **Partial Phase-2 write** | Eliminated by single-tx Phase 2 — any throw rolls all of Phase 2 back atomically | No recovery needed; the system is impossible to leave half-debited |
| **Retry exhaustion** | Laravel marks the job as failed after `tries=3` | Lands in `failed_jobs` table; `transfers:retry-failed <id>` re-queues it (operator-driven recovery). Tested by `DlqReplayTest` |
| **Ledger drift** (e.g. a row deleted by hand or by a buggy sweeper) | `ledger:reconcile` artisan command checks three invariants | Operator-driven; command exits non-zero with the offending transfer / account IDs |

### 4.2 What we don't recover from automatically

- **A transfer stuck in `PROCESSING` after retry exhaustion.** The DLQ replay command picks it up if the failure was on the queue side, but a transfer that committed Phase 1 and then never had Phase 2 attempted (e.g. worker container removed mid-handle) would sit at `PROCESSING` indefinitely. A "sweep `PROCESSING > N minutes` and re-dispatch" job is the correct next addition; we documented the gap rather than half-implement it.

### 4.3 Concurrency safety

Two correctness-critical pieces of behaviour:

- **`SELECT ... FOR UPDATE` ordered by UUIDv7 ASC.** All settlement transactions lock involved accounts in the same global order, eliminating circular-wait deadlocks.
- **Balance recomputed inside the lock.** No TOCTOU window — the balance we check is the balance we mutate.

A truly parallel test (forking processes) was punted in favour of a sequential test (`two_sequential_transfers_from_the_same_account_never_overdraw_it`) that proves the recompute-under-lock logic is correct. The `lockForUpdate()` directive itself is exercised only by Postgres at runtime, but `ledger:reconcile` will catch any drift even if the lock somehow didn't bite.

### 4.4 Reconciliation as a defence in depth

`php artisan ledger:reconcile` independently verifies three invariants:

1. **Per-transfer double-entry.** Every COMPLETED transfer must have exactly two ledger rows: one DEBIT, one CREDIT, of the same amount. Non-COMPLETED transfers must have zero ledger rows.
2. **Global zero-sum.** `SUM(amount * sign(direction))` across all `ledger_entries` is exactly 0.
3. **No user account has a negative balance.** Only the system account is allowed to go negative.

These checks pass on a healthy DB. Any drift (from a bug, manual hot-fix, restore-from-backup, etc.) gets surfaced with the offending transfer or account IDs. The command exits non-zero so it can be wired into a cron / monitoring pipeline.

### 4.5 How fail-vs-retry is actually decided

The processor differentiates "this will never succeed" from "this might succeed if we try again" by exception type, and the job class implements both outcomes:

- **`InsufficientBalanceException`** — a domain failure. The sender does not have the funds and that won't change by retrying. `ProcessTransferJob::handle()` catches it explicitly: it calls `markFailed()` (writes `status=FAILED`, `error_reason`, the `TransferFailed` audit row, and the current `attempts` count — all in one DB transaction) and then calls `$this->fail($e)`. The two are different things: `markFailed()` records *our* state, `$this->fail()` is Laravel's signal that retry is over and the job should land in `failed_jobs` immediately, skipping the remaining `tries` budget.
- **Anything else** (DB outage, Postgres deadlock victim, transient infrastructure error) — re-thrown unchanged. Laravel's queue worker catches it, increments the attempt counter, schedules a retry after `backoff = 5` seconds, and gives up after `tries = 3`. Final failure lands in `failed_jobs` and the operator-driven `transfers:retry-failed` command (§4.1) re-queues it via `queue:retry`.

The `transfers.attempts` column is populated inside `markFailed()` from `max(1, $this->attempts())` — Laravel's queue exposes the current attempt number, and the `max(1, …)` floor handles the synthetic "first attempt" case where the framework hasn't incremented yet. It exists so operators can see retry pressure on a row without joining against `failed_jobs`.

The two-tier idle hands `crash mid-handle` (the truly transient case) the right behaviour automatically: the connection drop at the DB layer rolls back Phase 2, the framework sees an unhandled `Throwable`, retries fire — and on re-attempt the idempotent Phase 1 + fresh Phase 2 either succeeds or burns the retry budget cleanly.

## 5. Scaling Strategy

### 5.1 Where the bottlenecks are

At our current shape, the dominant constraints in order:

1. **The `lockForUpdate` on hot accounts.** A high-throughput account becomes a serialization point — every transfer involving it queues behind every other. For a real trading platform you'd see this at popular merchant accounts.
2. **Single `queue:work` worker.** Sequential job processing.
3. **Database queue driver.** Polls the `jobs` table.
4. **Balance computation.** O(N) where N = number of ledger rows for that account.

### 5.2 Path forward

Roughly in increasing-cost order:

| Change | What it buys |
|---|---|
| Run multiple `queue:work` workers (already supported — just scale `worker` in compose) | Concurrent settlement; safe because of `FOR UPDATE` |
| Add a `account_balance_snapshots` table updated atomically with each ledger write | O(1) balance reads; ledger remains ground truth |
| Move from database queue driver to Redis or SQS | Higher dispatch throughput, better fanout, dedicated retry backoff |
| Partition `ledger_entries` by `account_id` (Postgres declarative partitioning) | Per-partition indexes; faster ledger scans |
| Shard the whole DB by `account_id` | Linear scale once a single Postgres can't keep up |
| Add a "stuck PROCESSING" sweeper command (cron) | Automatically recovers transfers that exhausted retries between Phase 1 and Phase 2 |
| Per-tenant queues / rate-limit buckets | Prevents one noisy tenant from starving others |

Most of these are well-understood patterns; none change the domain model.

### 5.3 Things that DO change at scale

- **Idempotency keys would become per-tenant.** Right now they're global, which is fine for a single API consumer but breaks if two unrelated callers happen to use the same key.
- **Multi-currency would arrive via per-account currency + an FX policy decision.** Either every transfer is same-currency only (simplest) or transfers carry a quote and an FX leg.
- **Audit retention.** `audit_logs` is unbounded in the current design. Production would archive older rows to cold storage with the same correlation_id.

## 6. Security & Compliance

### 6.1 Threat model

What the codebase is designed to be safe against:

- **Accidental misuse** — duplicate retries, double-clicks, malformed input, race conditions between legitimate concurrent callers, partial state from worker crashes.
- **Casual external actors** without an API key — any unauthenticated request is rejected before touching application logic.
- **Common web-app vulnerability classes** — SQL injection, mass assignment, IDOR via predictable identifiers, error-page information disclosure, validation bypass.

What it is **not** designed to be safe against (explicitly out of scope, see §6.12):

- An authenticated adversary with a valid API key.
- A compromised infrastructure layer (DB host, secrets store, log destination).
- Denial-of-service via legitimate-looking traffic above the per-key throttle.
- Insider threat from operators with direct DB access.

### 6.2 Authentication & authorization

- **Static API key in `X-Api-Key`** (`app/Http/Middleware/ApiKeyAuth.php`). SHA-256 hashed before comparison, with `hash_equals` for timing-safe equality so wrong-key responses don't leak via response-time differences.
- **Multiple keys supported** (comma-separated `API_KEYS` env). Lets different consumers be revoked independently without re-deploying for everyone.
- **Mass-assignment guard.** `Account` does not list `is_system` in `$fillable`, so the API cannot promote an account to system status by passing `is_system: true`. Verified by `CreateAccountTest::it_does_not_allow_setting_is_system_through_the_endpoint`.
- **System account is invisible to the API.** `GET /accounts/{id}`, `POST /deposits`, `POST /transfers` all 404 if the target is the system account. The system account exists for double-entry counterparty purposes only.
- **No authorization model beyond authentication.** Any valid key has full access to every endpoint and every account. Per-key scopes / per-tenant ownership of accounts are production-hardening work (§6.12).

### 6.3 Replay & idempotency

- **DB-enforced idempotency** via `UNIQUE` constraint on `transfers.idempotency_key`. Even under concurrent retries, the unique-violation race handler in `TransferService::deposit` and `initiateTransfer` catches `UniqueConstraintViolationException` and returns the original row. The pattern:
  1. Pre-check: `SELECT WHERE idempotency_key = ?`. Hit → return existing row, status 200.
  2. INSERT inside the write transaction.
  3. On unique-violation: re-fetch and return the race winner's row, status 200.
- **Defense against:** duplicate transfers from network retries, accidental double-submits, replay within the lifetime of a key.
- **Limitation:** idempotency keys are global. Two unrelated callers reusing the same key would observe each other's transfers. Production would scope by `(api_key_id, idempotency_key)`.

### 6.4 Input validation

- All write endpoints validate via Laravel `FormRequest` classes (`app/Http/Requests/Api/V1/`) at the boundary:
  - `name`: non-empty string ≤ 120 chars.
  - `amount`: integer ≥ 1.
  - `from_account_id` / `to_account_id`: valid UUID, `from` must differ from `to`.
  - `idempotency_key`: required string ≤ 120 chars.
- **Defense in depth at the DB layer** — `CHECK (amount > 0)`, CHECK constraints on enum-style columns, FK constraints on every relation. Even a hypothetical bypass of the FormRequest layer would be caught by Postgres.
- **Account UUIDs are looked up before any side effects**; missing accounts return 404 *before* the controller touches the service layer.

### 6.5 Injection prevention

- All queries go through Eloquent or `DB::table()` query builder with parameter binding.
- The two pieces of raw SQL (`Account::getBalance`, `ReconcileLedgerCommand::checkNoNegativeUserBalances`) bind parameters via `?` placeholders or `selectRaw` arguments — no string concatenation of user input.
- **Defense against:** OWASP A03 (Injection).

### 6.6 Concurrency safety

- **`SELECT ... FOR UPDATE` locks both involved accounts** in deterministic UUIDv7-ascending order, eliminating circular-wait deadlocks across concurrent settlements.
- **Sender balance recomputed under the lock** — no TOCTOU window between read and debit. The balance we check is the balance we mutate.
- **Two-phase processor** keeps Phase 2 (settlement) atomic; any throw rolls back all of Phase 2's writes.
- **Defense against:** double-spend, concurrent overdraft, lost updates.
- **Verified by:** `ProcessTransferJobTest::two_sequential_transfers_from_the_same_account_never_overdraw_it`. (A truly parallel test was punted in favour of the sequential one — see §4.3.)

### 6.7 Data integrity (DB-layer constraints)

The schema is hardened so that buggy code, accidental SQL writes, or a corrupt restore-from-backup cannot violate domain invariants without the DB rejecting the write:

- `transfers.amount > 0`, `ledger_entries.amount > 0` (CHECK).
- `transfers.type IN ('TRANSFER','DEPOSIT')`, `transfers.status IN (...)`, `ledger_entries.direction IN ('DEBIT','CREDIT')`, `audit_logs.event_type IN (...)` (CHECK).
- `transfers.idempotency_key` UNIQUE.
- `ledger_entries.transfer_id` NOT NULL (every ledger row has a parent operation).
- `accounts.is_system` partial unique index (DB rejects a second system account).
- All FKs declared with `restrictOnDelete()` or `nullOnDelete()` — orphan rows are impossible.

### 6.8 Audit trail & non-repudiation

- **Every state change writes an audit row in the same DB transaction as the change.** The audit log and the application state can never disagree — if you can see the state change, you can see the audit row.
- **Append-only by convention.** No code path UPDATEs or DELETEs `audit_logs`. The Eloquent model has no setters for mutating existing rows. Production hardening: a Postgres rule or `BEFORE UPDATE OR DELETE` trigger that raises an exception, making the table tamper-evident at the DB layer (currently soft).
- **`event_type` vocabulary fixed by CHECK constraint.** Typos can't sneak in.
- **`correlation_id` carried through every layer.** Either accepted from the client as `X-Correlation-Id` (if present and a valid UUID — useful when a caller wants to thread its own request ID through our logs) or generated server-side as a fresh UUIDv7 by `CorrelationId` middleware. It's then echoed back on the response, passed into the `ProcessTransferJob` constructor, pushed onto Log context inside `handle()`, and written to every `audit_logs` row. Reconstructing a transfer's full lifecycle is `WHERE correlation_id = ?` against `audit_logs` — and the same `correlation_id` is in every log line for the request, so logs and audit join cleanly.

### 6.9 Rate limiting & DoS

- `POST /transfers` is throttled per API key via Laravel's `RateLimiter`, default 60/min.
- Bucket key is the SHA-256 hash of the API key, set by `ApiKeyAuth` on the request — different keys have completely independent buckets.
- 429 responses include a `Retry-After` header.
- The throttle lives on the route, *after* `ApiKeyAuth` runs; an unauthenticated request always 401s before consuming a rate-limit slot, so the throttle can't be exhausted by anonymous traffic.
- **Limitation:** no global rate limit, no per-IP throttle, no concurrent-connection cap. A single key can saturate its 60/min ceiling; multiple keys can multiply that.

### 6.10 Information disclosure

- **`ForceJson` middleware** ensures every `/api/v1/*` response is JSON regardless of `Accept` header. Without it, validation errors on requests missing `Accept: application/json` returned an HTML redirect (potentially leaking framework chrome on some configurations).
- **API keys travel only in headers**, never in query strings — they don't end up in proxy access logs or browser history.
- **404 vs 403 ambiguity** for accounts you can't see (system account). We return 404 rather than 403; a real production system might also return 404 for accounts owned by a different tenant to avoid leaking existence.
- **Predictable identifiers.** UUIDv7 leaks creation time. For our domain (financial transfers, timestamps are public anyway) this is acceptable; for systems where id-leakage matters (e.g. healthcare records) UUIDv4 would be preferable.

### 6.11 Data protection at rest & in transit

- **No PII in the schema.** `accounts.name` is operator-supplied and untrusted. No email, no auth identity, no payment-card data.
- **Logs include amounts and account IDs.** Sensitive in some regulatory regimes. Production: ship logs to an access-controlled system (Loki, ELK, CloudWatch); consider redacting amounts at the application level.
- **Database credentials in plain `.env`.** Production would source these from a secrets manager (Vault, AWS SSM, GCP Secret Manager).
- **No TLS in the docker-compose stack.** Production must run behind an HTTPS terminator (nginx/Caddy/cloud load balancer).
- **No encryption at rest configured here.** Relies on Postgres + disk-level encryption configured at the infrastructure layer, not in this codebase.

### 6.12 Deliberate trade-offs (production hardening work)

Choices made knowingly. Each would close a real attack class but adds scope beyond an 8–12 hour exercise.

| Choice today | What it costs us | Production replacement |
|---|---|---|
| Static API key, no signing | A captured key is valid forever | HMAC over body + timestamp; reject stale or replayed requests |
| Keys live in `.env` | Plaintext on disk | Secrets manager with audit log |
| Every key has full scope | Scope creep over time, no separation of duties | `api_keys` table with per-key scopes (read-only / deposit-only / etc.) and a `kid` header for rotation without redeploy |
| HTTP only in dev compose | Sniffable in transit | Mandatory TLS via cloud LB |
| Global idempotency keys | Cross-tenant transfer leakage on key collision | Scope by `(api_key_id, key)` |
| Append-only audit by convention | A privileged DB session can still UPDATE/DELETE audit rows | Postgres rule or trigger that rejects UPDATE/DELETE; or write to a separate, append-only system (immudb, AWS QLDB) |
| No global rate limit | Aggregate DoS via many keys | Tiered limits (per-key, per-tenant, global) |
| Sequential overdraft test only | Lock contention itself is unverified | Forking-process or container-orchestrated parallel test |

### 6.13 Vulnerability classes deliberately out of scope

- **CSRF** — N/A; this is a stateless header-auth API, no cookies, no browser session.
- **Account lockout / fraud detection** — no anomaly detection, no velocity checks. A real wallet platform would have these.
- **KYC / AML hooks** — no identity, no sanctions screening, no suspicious-activity reporting.
- **GDPR right-to-erasure** — fundamentally hard with an append-only audit log. Production would need pseudonymization (replace user identifiers with random tokens, keep amounts). Out of scope here because we don't store user identities at all.
- **Secrets rotation** — no support for rotating API keys without restart.
- **Audit log retention policies** — `audit_logs` grows unbounded.
- **Side-channel attacks** beyond the timing-safe key comparison — not analysed.

### 6.14 Residual gaps a security review would surface

If a security audit ran today, these would be the credible findings:

1. **API-key compromise = full access.** No signing, no source-IP binding, no time-bound tokens, no scope limitations.
2. **Audit-log tampering possible at the DB layer.** Convention-based append-only is a soft guarantee; a SQL `UPDATE audit_logs ...` from a privileged session would succeed.
3. **No separation between API and worker DB roles.** Both run on the same connection with the same privileges; a compromise of either compromises everything.
4. **No structured PII handling.** None stored today, but no infrastructure (encryption, redaction) in place if we added it.
5. **Plaintext logs on the container filesystem.** A compromised container reveals all amounts and account IDs.
6. **Sustained legitimate-looking traffic within rate limits can still degrade service.** Single-key throttle doesn't bound aggregate worker load.

The short version: **the system is designed to be safe against accidents and concurrent legitimate users**. Defending against an authenticated adversary or a compromised infrastructure layer is documented production-hardening work — none of it is in the codebase, all of it is identified.

## 7. What we deliberately did not build

- **A "stuck PROCESSING" sweeper.** Documented in §4.2 — this is the natural next operational addition.
- **Cached `account_balance_snapshots`.** O(1) balance reads. No design questions, just engineering work; left out per the "don't over-engineer" steer.
- **Multi-currency.** Adds an FX policy decision on every transfer that crosses currencies. Out of scope.
- **A users / API-keys table.** The API-key list lives in env. A real deployment would put keys in a `api_keys` table with revocation timestamps and per-key scope.
- **OAuth / JWT.** Would replace the static API key with per-tenant credentials.
- **Webhook outbound notifications.** A common production add — when a transfer hits COMPLETED or FAILED, fire a webhook to the originating tenant. Easily added by listening to the relevant audit-log event.
- **Forking-process concurrency tests.** The `lockForUpdate` directive is in the code; sequential tests prove the recompute-under-lock logic; reconciliation catches drift if anything else is wrong. A truly parallel test would be belt-and-braces but punted for the time budget.
